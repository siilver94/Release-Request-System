# CLAUDE.md — 릴리즈 상태 요청 시스템

> 이 앱 작업을 시작할 때 **가장 먼저** 읽는 파일. 두 번째로 `_README.md`를 읽는다.

## 이 앱이 하는 일 (한 줄)
연구소 연구원이 PLM Data Cleansing / 설계 수정을 위해, 특정 **품번**의 릴리즈 상태 변경을 **DX팀**에 요청하고, DX팀이 처리 → 요청자가 최종 확인하는 요청·처리 관리 시스템.

## 업무 흐름
```
요청자(연구원)                     DX팀 담당자
  │ 품번 + 도면종류(3D/2D/WTPart 다중) 작성 후 요청
  ▼
[진행중 · 요청됨]  ── 메일→DX팀
  │ DX팀이 PLM 상태 변경 작업 후 "수정완료" 클릭
  ▼
[진행중 · 수정완료] ── 메일→요청자
  │ 요청자 "작업완료" 클릭
  ▼
[완료 · 작업완료]  ── 메일→DX팀

 ※ DX팀 "강제완료"(요청자가 오래 미확인) → [완료 · 강제완료] ── 메일→요청자
```
- **승인 단계 없음.** 요청되면 바로 `진행중`.
- 컬럼은 **진행중 / 완료 2개만** (원본의 "시작전" 컬럼 제거).

## 사용자 역할 (3종)
| 역할 | 하는 일 |
|---|---|
| 요청자(연구원) | 요청 등록, 본인 건의 "작업완료" 클릭 |
| DX팀 담당자 | "수정완료" / "강제완료" / 일괄·선택 강제완료 |
| 관리자 | 전체 조회 (varIsViewer 로 판별) |

## 원본 기준 (중요)
- 이 앱은 **시작실 작업의뢰 시스템**(`WorkRequest_Screen.yaml.txt`)을 개조해서 만든다.
- 원본은 승인 흐름(팀장 승인/작업자 반려, `ApprovalState` 필드, 시작전 컬럼)이 있으나 **이 앱에서는 모두 제거**한다.
- 변수/필드명이 아직 작업의뢰서 기준(`das`, `제목`, `RequestQty` 등)으로 남아있음 → 개조하며 릴리즈 도메인에 맞게 교체.

## 가장 중요한 규칙: 파일 != 원본
- source of truth는 실제 Power Apps 앱. 폴더의 `.yaml.txt`는 "같이 보는 작업본".
- 파일을 고쳐도 **주인이 Power Apps에 붙여넣기 전까지 실물에 반영 안 됨.** "파일 고쳤으니 끝"이라고 하지 말 것.

## 코드 수정 워크플로우 (반드시 이 순서)
1. 주인이 "이 코드로 반영하자"고 하면
2. **먼저 diff로 보여준다** (어느 파일/컨트롤/속성, 변경 전->후). 관련 없는 부분 절대 건드리지 않음(전체 재작성 금지).
3. 주인이 "반영해줘" 승인 -> 그때 파일 수정
4. 수정 후 **Power Apps에 붙여넣을 코드 블록**을 따로 제시
5. 주인이 "반영 완료" -> 사이클 종료
6. 구조 변경(스크린/변수 추가 등)이면 `_README.md`도 함께 업데이트 제안

## 라이선스 제약 (무료 플랜)
- Dataverse 불가 -> **SharePoint List만** 사용
- Power Automate는 표준 커넥터만 (프리미엄 금지)
- 유료가 최선이면 "이건 유료"라고 먼저 말하고 무료 대안 제시

## 앱 밖(별도 flow)에서 처리하는 것
- **3일 경과 리마인드 메일**: 앱은 열려야만 동작하므로 불가. Power Automate **스케줄(반복) flow**로 매일 조건 검사 후 발송. (표준 커넥터: SharePoint 조회 + Outlook 발송 -> 무료 OK)

## 이메일 / 상태 변경 방식 (원본 재사용)
- 이메일: `Office365Outlook.SendEmailV2` (표준 커넥터, 무료 OK)
- 상태 변경: `Patch(리스트, 레코드, {Stage:..., Status:...})`

## 성능
- delegation 가능 함수 우선, 불가 시 미리 경고
- Collection 캐싱으로 체감 속도 확보
- 반복 로직 중복 지적, 복잡 수식은 단순화 먼저 제안

## UI 톤
- 회사 업무용, Fluent 기반 차분한 톤. Teams 데스크톱·모바일 둘 다 고려.
- 원본의 헤더/카드/배지 스타일 유지, 도메인 텍스트만 교체.
